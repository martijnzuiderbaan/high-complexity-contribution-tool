# Private Beta Release Notes

## 1. Release Summary

This private-beta package is a Markdown-first release of the **High-Complexity
Contribution Tool** / **Instrument Complexiteit naar Bijdrage**.

It is for trusted reviewers, high-complexity adults, coaches, facilitators,
managers, HR partners, and product/commercial reviewers who can respect the
current boundaries and provide product-focused feedback.

It is not a public launch, validated assessment, IQ test, giftedness diagnosis,
psychometric instrument, hiring tool, promotion tool, performance-management
tool, employee-ranking tool, clinical tool, occupational-health tool, legal
tool, or production software application.

The intended output is a clearer role-design conversation and, where safe, one
bounded 30-day work experiment. It is not a conclusion about a person.

## 2. Release Status

Current status:

- Private beta.
- Invite-only and manually operated.
- Static site plus Markdown documentation.
- No live data collection through the site.
- No accounts, authentication, backend, database, external analytics,
  tracking scripts, cookies, payments, checkout, or real form submission.
- No public deployment or public support process has been completed.
- Privacy, security, and terms materials are placeholders and planning docs,
  not final reviewed public materials.

Private individual beta is conditionally suitable only when the maintainer uses
the direct private contact route and manual records SOP described in
`docs/PRIVATE_BETA_RECORDS_SOP.md`.

Employer pilots, paid pilots, and public launch remain gated.

## 4. Recommended Use Order

For an individual adult:

1. Read `docs/BETA_DISCLOSURE_AND_CONSENT.md` and the relevant boundaries in
   `README.md`.
2. Use `docs/USER_GUIDE.md` to choose a narrow path.
3. Complete only the relevant private sections of `docs/WORKBOOK_v1.md`.
4. Use `docs/INTERPRETATION_GUIDE_v1.md` to keep interpretation qualitative
   and cautious.
5. Share only a manager-safe summary if sharing is safe and useful.

For a coach or facilitator:

1. Read `docs/RESPONSIBLE_USE_GUIDE.md`.
2. Review `docs/ONBOARDING.md` and `docs/DEMO_FLOW.md`.
3. Use `docs/WORKBOOK_v1.md`, `docs/INTERPRETATION_GUIDE_v1.md`, and
   `docs/MANAGER_CONVERSATION_GUIDE_v1.md` together.
4. Use the examples to test whether the flow is clear before using live
   participant material.
5. Keep feedback product-focused and do not send client workbook content.

For a manager or HR reviewer:

1. Read the non-selection and employer boundaries in `README.md`,
   `docs/RESPONSIBLE_USE_GUIDE.md`, and `docs/MANAGER_CONVERSATION_GUIDE_v1.md`.
2. Use manager-facing materials to review role/context conditions only.
3. Do not request completed private workbook content.
4. Do not store workbook content in HR files.
5. Treat any real use as voluntary, bounded, and role-design oriented.

For a beta reviewer:

1. Read this release note and `docs/DOCUMENTATION_INDEX.md`.
2. Review `docs/CONTENT_QA_REPORT.md`, `docs/BETA_PLAN.md`,
   `docs/PRIVATE_BETA_RECORDS_SOP.md`, and `docs/OPEN_ITEMS.md`.
3. Preview the static site from `site/README.md`.
4. Walk through `docs/DEMO_FLOW.md`.
5. Send feedback on clarity, boundaries, missing artifacts, and readiness gaps.

For a product or commercial reviewer:

1. Read `docs/PRD_v1.md`.
2. Read `docs/MONETIZATION.md`.
3. Read `docs/ACCESS_AND_ACCOUNTS_PLAN.md`.
4. Read `docs/ANALYTICS_AND_MEASUREMENT.md`.
5. Read `docs/MARKETING.md`, `docs/ROADMAP.md`, and

## 5. Safe-Use Boundaries

This release must stay within these boundaries:

- Non-diagnostic: it does not diagnose giftedness, neurodivergence, burnout,
  mental health, disability, or any clinical or occupational-health condition.
- Non-selection: it is not for hiring, screening, promotion, succession,
  high-potential programs, employee ranking, or support-entitlement decisions.
- Non-performance-management: it must not be used for performance ratings,
  discipline, firing, formal HR decisions, or personnel-file characterization.
- No mandatory employer use: employer-side use must be voluntary, bounded, and
  role/context oriented.
- No sensitive data in the static site: the page has no input fields, no
  backend, no storage, no mailing-list integration, and no real submission.
- Adult control: the adult decides what private information is kept private or
  shared.

## 6. Known Limitations

- The tool is not validated as a psychometric, diagnostic, clinical,
  occupational-health, legal, HR, or commercial outcome instrument.
- There is no automated intake.
- There is no final legal/privacy review.
- There is no payment flow, checkout, invoice automation, refund process, or
  license delivery process.
- There is no account system, dashboard, portal, database, backend API, or file
  upload.
- There is no production deployment.
- There is no external analytics provider.
- Manual tracking only is available for private beta operations.
- Facilitator scripts, a separate tracker/review handout, employer pilot
  scope, and public-ready terms/privacy/security materials remain open work.

- Names or identifiable details of employees, managers, colleagues, clients,
  candidates, organizations, teams, or family members.
- Employer records, screenshots, exports, or confidential files.

## 8. Private Beta Entry Criteria

Before someone is invited, all of the following should be true:

- The use case is voluntary, role/context oriented, and appropriate for the
  current beta wave.
- The participant has a direct private contact route agreed outside the static
  site.
- The maintainer can keep contact records separate from product-learning notes.
- The participant understands that completed scans and sensitive details should
  not be sent as feedback.
- The participant understands that the tool is non-diagnostic and not for HR
  decisions.

Docs to send or link before use:

- `docs/BETA_DISCLOSURE_AND_CONSENT.md`
- `docs/RESPONSIBLE_USE_GUIDE.md`
- `docs/PRIVACY_PLACEHOLDER.md`
- `docs/SECURITY.md`
- `docs/TERMS_AND_USE_PLACEHOLDER.md`
- The relevant current v1 package documents from `docs/DOCUMENTATION_INDEX.md`

Consent or disclosure acknowledgement needed:

- Non-diagnostic boundary acknowledged.
- No hiring, promotion, performance, ranking, diagnosis, or support-entitlement
  use acknowledged.
- No completed scans or sensitive details will be sent as feedback.
- Quote or testimonial permission remains separate and defaults to no.

Manual records to keep:

- Minimal contact record.
- Consent/disclosure log.
- Product-learning tracker without personal case detail.
- Deletion/correction status if relevant.
- Weekly beta summary without identifiable stories.

Use `docs/PRIVATE_BETA_RECORDS_SOP.md` for the operational details.

## 9. Exit Criteria

Move to more beta users only if:

- Participants understand the boundaries.
- Feedback stays product-focused.
- No unmanageable sensitive disclosures or misuse patterns appear.
- The workbook path becomes clearer rather than more confusing.
- Open private-beta issues are logged or resolved.

Move to facilitated beta only if:

- Individual/private use shows recognizable value and manageable risk.
- Coaches or facilitators can follow the materials without author explanation.
- Pause, decline, referral, and accidental-disclosure handling are clearer.
- A separate tracker/review handout or facilitation script is ready if needed.

Move to an employer pilot only if:

- A written employer pilot scope exists.
- Participation is voluntary and declining has no adverse consequence.
- Adult control over sharing is explicit.
- Completed workbook content stays out of HR files.
- Data boundaries, stop conditions, review criteria, and process ownership are
  documented.

Explore paid pilots only if:

- One paid package is clearly defined.
- Terms, license scope, privacy, refund/cancellation, support, and delivery
  boundaries can be handled manually.
- Buyers accept the non-diagnostic and responsible-use constraints.
- Pricing interest is specific enough to test, not only encouraging feedback.
